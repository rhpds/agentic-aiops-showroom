== Key concepts to carry forward

*Context isolation is an architecture decision, not an optimization*

Subagents don't share context windows — this is why findings stay clean and supervisor reasoning stays reliable.
When you design multi-agent systems, think first about context boundaries, not tool lists.

*The prompt is the policy*

`risk_classifier.py` is code, but `prompts/supervisor.txt` is policy.
The instructions you give an agent — what it must do, must not do, and must ask about — are your compliance surface.
They require the same governance as any other policy document at Meridian Financial.

*Model selection is a portfolio decision*

You used five different models across six modules.
No single model is right for every task.
`gpt-oss-20b` for speed. `gpt-oss-120b` for reasoning. `qwen3-235b` for complex analysis. `gemini-2.5-pro` for long context.
Red Hat MaaS gives you access to the portfolio; your agent logic makes the selection.

*Autonomy is a spectrum, not a switch*

Human-in, human-on, human-out — these are distinct patterns for distinct risk levels.
The engineering work is building the classifier that routes correctly.
The governance work is deciding what goes in each bucket.
Both matter.

*MTTR improvement comes from the investigation phase, not the execution phase*

Your agent reduced time-to-root-cause and time-to-compliance-report dramatically.
Remediation execution time (re-running the AAP job) is the same.
Set expectations accordingly when presenting this to stakeholders.
