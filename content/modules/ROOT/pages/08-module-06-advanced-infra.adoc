Three Kira tickets.
Two infrastructure layers.
One cascading failure.

Meridian Financial's Friday compliance window starts in four hours.
The FCA auditors will check change records, incident logs, and remediation evidence.
Everything must be documented.
Everything must be traceable.

Your agent knows how to triage. It knows how to investigate. It knows when to ask for help.
Now it faces the kind of incident that separates a useful tool from a reliable system: a multi-hop failure where fixing one problem reveals another, and the blast radius is unclear until you've dug three layers deep.

In this module, you'll put the full architecture to work — supervisor, subagents, risk classifier, approval gate, audit trail — against a realistic incident scenario.
You'll also measure what you've built against the MTTR and SLA targets Meridian set at the start.

== Learning objectives

By the end of this module, you'll be able to:

* Orchestrate a full deep agent investigation across VM and OpenShift infrastructure layers
* Handle cascading failure scenarios where root cause discovery triggers additional investigations
* Generate a compliance-ready incident report from agent findings and audit trail data
* Evaluate the framework's strengths and limitations against the original Meridian Financial requirements

== The scenario: A cascade begins

At 02:15 UTC, three Kira tickets open in rapid succession:

----
KIRA-0047 | MEDIUM | SSH_AUTH_FAILURE       | prod-finance-east | RHEL VMs 3/12
KIRA-0048 | HIGH   | OCP_DEPLOYMENT_FAILED  | oc-cluster-01     | payment-processor pod
KIRA-0049 | HIGH   | ANSIBLE_VARS_MISSING   | prod-finance-east | credentials.yml absent
----

These look like three separate incidents.
They are not.

Your agent will discover — as you investigate alongside it — that:

* KIRA-0049 is the root cause: `credentials.yml` was deleted during a playbook refactor pushed to Gitea at 02:09 UTC
* KIRA-0047 is a downstream effect: SSH auth fails because the key material referenced in `credentials.yml` is no longer accessible to the playbook
* KIRA-0048 is a further downstream effect: the `payment-processor` deployment in OpenShift failed because its `ConfigMap` is populated by an Ansible job that depends on `credentials.yml`

One deleted file. Three open tickets. Four hours to the compliance window.

== Troubleshooting

*Issue*: `sre_ocp_agent` cannot find `payment-processor` deployment
*Solution*: Confirm the deployment exists in your namespace: `oc -n {aap2_organization} get deployments`.
If absent, the facilitator may need to trigger the OCP failure scenario.

*Issue*: Gitea commit history doesn't show the credentials.yml deletion
*Solution*: The test commit may need to be set up before the lab session.
Use `git log --all -- vars/credentials.yml` in a local clone of the repo.
If nothing appears, contact your facilitator — the scenario may need to be re-triggered.

*Issue*: Supervisor detects three tickets but doesn't group them as a cascade
*Solution*: Add a more explicit correlation instruction to `prompts/supervisor.txt`:
"When multiple tickets open within a 10-minute window, investigate whether they share a common upstream cause before treating them independently."

*Issue*: Incident report is missing the cascade timeline section
*Solution*: The timeline section requires timestamps from both Gitea (commit time) and Kira (ticket open time).
Verify both `sre_playbook_agent` and `sre_job_log_agent` return timestamps in their findings.
Check `prompts/supervisor.txt` to ensure the report prompt asks for a timeline section explicitly.

== Takeaways

*What you demonstrated in this module*:

* Multi-ticket correlation: the agent identified a three-ticket cascade from a single root cause
* Cross-layer investigation: VM (SSH auth), OpenShift (deployment failure), and source control (Gitea commit) in one coordinated run
* Automated compliance report generation from agent findings and audit trail data
* Measurable MTTR improvement — from ~45 minutes to ~8–10 minutes for the investigation and documentation phases

*Honest assessment — where the framework excels*:

* Rapid investigation across heterogeneous data sources (job logs, playbooks, VM state, OCP events)
* Consistent, repeatable triage logic — no cognitive load variation between 3pm and 3am
* Automatic audit trail — every agent decision is logged without extra effort
* Scalable — adding a new infrastructure layer means adding a new subagent, not retraining the whole system

*Honest assessment — where the framework struggles*:

* Novel failure modes: the agent excels at pattern-matching known error classes; genuinely new failure modes require human insight to recognize
* Tool reliability: each subagent tool call is a potential failure point; brittle tool implementations break the chain
* Model non-determinism: `temperature=0` helps, but LLMs can still produce different outputs on identical inputs — not ideal for compliance contexts where reproducibility matters
* Complexity: a four-subagent supervisor with risk classification, approval gates, and compliance logging is a significant system to maintain

*The second-generation agentic difference*:

First-generation agents (chatbots, RAG systems) answer questions.
Second-generation agents (what you built) *act* — they invoke tools, accumulate state, make decisions, and stop at the right moments.
The Deep Agents framework provides the orchestration; your job was to provide the operational logic, the compliance rules, and the judgment about when automation is appropriate.

That judgment — encoded in `risk_classifier.py` and `prompts/supervisor.txt` — is the most valuable thing you built.
Not the code. The decisions.
