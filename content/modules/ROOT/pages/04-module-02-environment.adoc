= Module 2: Your Environment
:source-highlighter: rouge
:toc: macro
:toclevels: 1

Before you write a single line of agent code, you need to know your platform.

A misconfigured API key, an unreachable service, or a misunderstood topology will cost you more time than any LLM reasoning error.
In this module, you'll walk every component of your environment — AAP, Gitea, RocketChat, Kira, the RHEL VM target, and your OpenShift namespace — and verify that they connect the way your agent will expect.

You'll also get your first look at Red Hat MaaS: what models are available, how authentication works, and how your agent will call it — all via API, with no browser UI required.

toc::[]

== Learning objectives

By the end of this module, you'll be able to:

* Describe the topology of your lab environment and how each component connects
* Verify connectivity to all services your agent will interact with
* Explain how Red Hat MaaS exposes language models via LiteLLM and how your virtual key controls access
* Identify the environment variables your agent code will use throughout this workshop

== Exercise 2: Verify service connectivity

From the Showroom terminal, run the full connectivity check:

[source,bash]
----
# 1. OpenShift — confirm namespace access
oc project {aap2_organization}
oc get pods

# 2. Gitea — confirm API access
curl -s {gitea_url}/api/v1/user \
  -u {gitea_username}:{gitea_password} \
  | python3 -c "import sys,json; d=json.load(sys.stdin); print(f'Gitea user: {d[\"login\"]}')"

# 3. Red Hat MaaS — list available models
curl -s {litellm_api_base_url}/models \
  -H "Authorization: Bearer {litellm_virtual_key}" \
  | python3 -c "
import sys, json
data = json.load(sys.stdin)
print('Available models:')
for m in data.get('data', []):
    print(f'  - {m[\"id\"]}')
"

# 4. RHEL VM — confirm SSH connectivity (via cluster network)
ssh -o StrictHostKeyChecking=no \
    {rhel_vm_user}@{rhel_vm_hostname} \
    'echo "RHEL VM: $(hostname) — $(uptime -p)"'
----

== Exercise 4: Set your environment variables

Your agent code will read configuration from environment variables.
Set them now so they're available for all subsequent modules.

[source,bash]
----
# Create your environment file (do NOT commit this to Gitea)
cat > ~/agent-env.sh << 'EOF'
# Red Hat MaaS (LiteLLM)
export MAAS_API_BASE="{litellm_api_base_url}"
export MAAS_API_KEY="{litellm_virtual_key}"

# Kira ticket system
export KIRA_URL="{kira_url}"
export KIRA_USER="{user}"
export KIRA_PASSWORD="{password}"

# RocketChat
export ROCKETCHAT_URL="{rocketchat_url}"
export ROCKETCHAT_USER="{user}"
export ROCKETCHAT_PASSWORD="{password}"

# Gitea
export GITEA_URL="{gitea_url}"
export GITEA_USER="{gitea_username}"
export GITEA_PASSWORD="{gitea_password}"

# AAP
export AAP_ORG="{aap2_organization}"
export AAP_USER="{aap2_username}"
export AAP_PASSWORD="{aap2_password}"

# RHEL VM
export RHEL_HOST="{rhel_vm_hostname}"
export RHEL_USER="{rhel_vm_user}"
export RHEL_PASSWORD="{rhel_vm_password}"
EOF

# Load variables into your current session
source ~/agent-env.sh

# Verify
echo "MaaS base: $MAAS_API_BASE"
echo "Kira URL:  $KIRA_URL"
echo "AAP org:   $AAP_ORG"
----

IMPORTANT: The `agent-env.sh` file contains credentials. It is created in your home directory on the Showroom terminal and is not persisted to Gitea. Do not share it or commit it.

=== Verify

✓ `agent-env.sh` is created and sourced without errors

✓ `echo $MAAS_API_BASE` returns `{litellm_api_base_url}`

✓ `echo $KIRA_URL` returns `{kira_url}`

== Troubleshooting

*Issue*: `curl` to MaaS returns `401 Unauthorized`
*Solution*: Your virtual key may have expired (keys are valid for {litellm_key_duration}).
Contact your facilitator for a new key.

*Issue*: SSH to RHEL VM times out
*Solution*: The RHEL VM is accessible only from within the cluster network via the Showroom terminal.
Confirm you are using the Showroom terminal, not a local terminal.
Hostname: `{rhel_vm_hostname}`.

*Issue*: `oc get pods` returns nothing
*Solution*: Some pods may not be running yet if your environment was recently provisioned.
Wait 2 minutes and retry. If still empty, run `oc get all` to see what exists.

*Issue*: Gitea returns `{"errors":[{"message":"...","locations":[...]}]}`
*Solution*: You may be hitting the API at a GraphQL endpoint. Use the REST API path `/api/v1/user` as shown in the exercise.

== Takeaways

*What you established in this module*:

* Your agent operates across five services: AAP, Kira, RocketChat, Gitea, and Red Hat MaaS
* Red Hat MaaS provides access to six models via the LiteLLM API — no browser access needed
* Model selection is a deliberate choice: small/fast for triage, large/capable for reasoning, long-context for full log analysis
* Environment variables are the clean, portable way to configure your agent — and keeping credentials out of source control is non-negotiable

*The MaaS advantage*:

In a regulated environment like Meridian Financial, using externally hosted commercial AI APIs creates data residency and privacy risks — job logs may contain hostnames, credentials, and system topology.
Red Hat MaaS runs within Red Hat's infrastructure, giving you access to both OSS and Frontier models through a single, governed API endpoint with per-user key management and usage tracking.
