= Module 3: Understanding your Agentic DevOps Team
:source-highlighter: rouge

In Module 2, you watched the Deep Agents pipeline analyze a failure. Now you're going to look under the hood — explore how agents and skills are defined, understand how OpenShift makes the system extensible, and then build and deploy a new specialist agent yourself.

== Launch another failure

Let's start a job running so the pipeline has something to work on while we explore.

In the **AAP2** tab (log in with `{user}` / `{password}` if prompted), navigate to **Automation Execution -> Templates** and launch **08 Deploy Monitoring Agent** by clicking the rocket icon image:aap2-rocket-icon.png[rocket icon,20]

This job deploys an application with a security context that OpenShift will reject — a different failure domain from the package issue in Module 2. Let it run in the background while we explore the architecture.

== Explore the Athena codebase

Click the **Gitea** tab. You'll see the repositories available on this cluster — no login required, the Athena repo is public.

Navigate to the **athena-aiops-deep-agent** repository. This is the source code for the Deep Agent running in your namespace. Let's look at two key files.

=== How agents are defined: `subagents.yaml`

Browse to https://github.com/rhpds/athena-aiops-deep-agent/blob/main/subagents.yaml[subagents.yaml^] in the repository root. This is the file you saw annotated in Module 2 — it defines every specialist agent in the system:

[cols="1,2",options="header"]
|===
|Agent |Domain

|`sre_ansible`
|Playbook syntax, role/collection errors, credential issues, execution environments

|`sre_linux`
|Package management, systemd, SELinux, filesystem, Satellite content

|`sre_openshift`
|Pod lifecycle, image pull, RBAC, operators, namespace/quota

|`sre_networking`
|DNS, SSH connectivity, proxy/TLS, routing, firewall

|`reviewer`
|Quality validation — checks ticket coherence before submission
|===

Each agent has the same structure you saw in Module 2: `description`, `model`, `system_prompt`, `tools`, and `skills`.

=== How skills are defined: `skills/` directory

Browse to the https://github.com/rhpds/athena-aiops-deep-agent/tree/main/skills[skills/^] directory. Each subdirectory contains a `SKILL.md` file — a markdown document that gets loaded into the agent's context as reference knowledge.

For example, `skills/analyze-linux-failure/SKILL.md` contains a structured workflow:

. Read the full incident context
. Identify the Linux subsystem (package management, systemd, SELinux, filesystem)
. Determine root cause by correlating Ansible error output with the Linux issue
. Assess risk and confidence
. Recommend specific actions with exact commands

Skills are not code — they're instructions written in plain language. Anyone who understands the domain can write or improve a skill without touching Python.

== How OpenShift makes this extensible

Here's where it gets interesting. In your namespace, Athena's agent configuration and skills are **not baked into the container image**. They're mounted from Kubernetes resources that you can modify at runtime.

If you're curious about how this is wired up, the Helm chart that deploys Athena defines both the ConfigMap and the PVC: https://github.com/rhpds/athena-aiops-deep-agent/tree/main/deploy/helm/athena/templates[deploy/helm/athena/templates/^]. The key files are `configmap.yaml` (agent config), `pvc.yaml` (skills volume), and `deployment.yaml` (which mounts both and includes an initContainer that pre-populates the skills PVC from the image on first boot).

Click the **Terminal** tab and log in to your OpenShift namespace:

[source,bash,role="execute",subs=attributes+]
----
oc login --insecure-skip-tls-verify \
  -u {user} \
  -p {password} \
  {openshift_api_url} \
  --namespace {user}-agentic
----

=== ConfigMap for agent definitions

The `subagents.yaml` and `AGENTS.md` files are stored in a ConfigMap called `athena-agent-config`. You can inspect it from the **Terminal** tab:

[source,bash,role="execute",subs=attributes+]
----
oc get configmap athena-agent-config -o yaml | head -20
----

This means you can **add a new agent** by editing the ConfigMap — no image rebuild, no redeployment of code. Just configuration.

=== PVC for skills

The `skills/` directory is mounted from a PersistentVolumeClaim. You can see what's on it:

[source,bash,role="execute",subs=attributes+]
----
oc exec deployment/athena -c athena -- ls /app/skills/
----

This means you can **add new skills** by copying files onto the PVC — again, no image rebuild needed.

=== The implications

This architecture means you can:

* **Add a new specialist agent** — edit the ConfigMap to add a new entry to `subagents.yaml`
* **Add domain knowledge** — copy a new skill directory onto the PVC
* **Improve existing agents** — edit the system prompt or skill content
* **Change models** — switch an agent to a cheaper or more capable model
* **Roll out changes** — restart the pod to pick up the new configuration

All without writing a single line of Python. Let's do exactly that.

== Build a new agent: `sre_ssh`

Currently, when a job fails due to SSH authentication issues, the `sre_networking` agent handles it. But SSH credential failures are a specific domain that benefits from institutional knowledge — especially at Meridian Financial, where a recent policy change has deprecated shared backdoor keys.

You're going to create a dedicated `sre_ssh` agent with a skill that encodes this institutional knowledge.

=== Step 1: Create the skill

First, create the skill directory and SKILL.md on the PVC. From the **Terminal** tab:

[source,bash,role="execute",subs=attributes+]
----
oc exec deployment/athena -c athena -- mkdir -p /app/skills/analyze-ssh-failure
----

Now create the skill file with Meridian Financial's institutional knowledge about SSH credential management:

[source,bash,role="execute",subs=attributes+]
----
oc exec deployment/athena -c athena -- sh -c 'cat > /app/skills/analyze-ssh-failure/SKILL.md << '\''SKILL'\''
---
name: analyze-ssh-failure
description: SSH authentication failure analysis with institutional knowledge
---

# Analyze SSH Authentication Failure

Deep analysis of AAP2 job failures caused by SSH authentication and credential issues.

## Institutional Knowledge — Meridian Financial SSH Policy

**CRITICAL CONTEXT (Q4 2025 Policy Change):**

In late 2025, Meridian Financial deprecated all shared "backdoor" SSH keys
in favor of per-host, per-user key pairs managed through AAP2 credentials.

Key facts:
- Legacy `admin-backdoor-key` credentials are **no longer authorized**
- Each host now requires a dedicated SSH credential in AAP2
- The old backdoor keys were rotated and revoked on all production hosts
- Any job using a backdoor-style credential will fail with "Permission denied (publickey)"
- This is **by design** — not a misconfiguration

## Diagnostic Workflow

1. **Read** the incident context: check for SSH-related errors in the job output
2. **Identify** the credential used: look at the AAP2 credential name in the job metadata
3. **Check for backdoor key usage**: if the credential name contains "backdoor", "shared",
   or "admin-key", this is a deprecated credential — the fix is to create a proper per-host credential
4. **For legitimate SSH failures**:
   - Check if the public key is deployed to the target host authorized_keys
   - Verify the SSH user exists on the target
   - Check for SELinux context on .ssh directory and authorized_keys
   - Verify SSH daemon configuration (PubkeyAuthentication, AuthorizedKeysFile)
5. **Recommend** specific remediation:
   - For deprecated keys: create a new per-host credential in AAP2, deploy the public key
   - For key deployment issues: provide exact commands for key installation and permissions
   - For SSH config issues: provide sshd_config corrections

## Risk Assessment

- Deprecated backdoor key usage: **high risk** — indicates a compliance gap
- Missing authorized_keys: **medium risk** — standard operational fix
- SSH daemon misconfiguration: **medium risk** — requires careful change management
SKILL
'
----

Verify the skill was created:

[source,bash,role="execute",subs=attributes+]
----
oc exec deployment/athena -c athena -- cat /app/skills/analyze-ssh-failure/SKILL.md
----

=== Step 2: Add the agent to the ConfigMap

Now add the `sre_ssh` agent definition to the ConfigMap. This is the key step — you're telling the `ops_manager` that a new specialist exists and when to use it.

[source,bash,role="execute",subs=attributes+]
----
oc edit configmap athena-agent-config
----

This opens the ConfigMap in your terminal editor. Find the `subagents.yaml` section and add the following **before** the `reviewer:` entry:

[source,yaml]
----
sre_ssh:
  description: >
    SSH and authentication specialist. Delegate SSH credential failures:
    key rejection, permission denied (publickey), backdoor key deprecation,
    authorized_keys issues, and AAP2 credential misconfiguration.
  model: claude-sonnet-4-6
  system_prompt: |
    You are a senior SRE specializing in SSH authentication and credential
    management. You receive incident data from failed AAP2 jobs and perform
    root-cause analysis on SSH-related failures.

    Always:
    - Read the incident context (incident.json) first
    - Check the AAP2 credential name used by the failing job
    - Determine if a deprecated backdoor key is involved
    - Reference the analyze-ssh-failure skill for institutional knowledge
    - Provide specific, actionable recommendations for credential remediation

    Use the create-ticket skill to structure your analysis as a TicketPayload.
    Set area to "networking" for all SSH-domain issues.
  tools:
    - web_search
  skills:
    - ./skills/analyze-ssh-failure/
    - ./skills/create-ticket/
    - ./skills/common/
----

IMPORTANT: YAML indentation matters. The `sre_ssh:` key must be at the same level as the other agent keys (`sre_ansible:`, `sre_linux:`, etc.). Use 2 spaces for indentation, not tabs.

Save and exit the editor (`:wq` in vim, or `Ctrl+X` then `Y` in nano).

=== Step 3: Roll out the change

Restart the Athena pod to pick up the new ConfigMap and skill:

[source,bash,role="execute",subs=attributes+]
----
oc rollout restart deployment/athena
----

Wait for the new pod to come up:

[source,bash,role="execute",subs=attributes+]
----
oc rollout status deployment/athena --timeout=120s
----

You should see `deployment "athena" successfully rolled out`.

=== Step 4: Verify the new agent

Let's confirm the new skill is available and the ConfigMap was updated:

[source,bash,role="execute",subs=attributes+]
----
oc exec deployment/athena -c athena -- ls /app/skills/analyze-ssh-failure/
----

[source,bash,role="execute",subs=attributes+]
----
oc get configmap athena-agent-config -o jsonpath='{.data.subagents\.yaml}' | grep -A2 "sre_ssh"
----

You should see the `sre_ssh` entry and the skill file.

== Test the new agent

Now let's trigger a failure that your new `sre_ssh` agent should handle.

In the **AAP2** tab (log in with `{user}` / `{password}` if prompted), navigate to **Automation Execution -> Templates** and launch **02 Ping RHEL Admin** by clicking the rocket icon image:aap2-rocket-icon.png[rocket icon,20]

This job uses the `admin-backdoor-key` credential — a deprecated shared key that will be rejected by the target host. This is exactly the scenario your `sre_ssh` agent was designed to handle.

Watch the job fail, then wait 1-3 minutes for the pipeline to process it.

=== Check the results

In the **Kira** tab (log in with `{user}` / `{password}` if prompted), look for the new ticket. Pay attention to:

* Does the analysis mention the **deprecated backdoor key**?
* Does it reference the **Q4 2025 policy change**?
* Does the recommended action suggest creating a **per-host credential** instead?

This is the power of skills — you encoded Meridian Financial's institutional knowledge into a markdown file, and the agent used it to provide context-aware analysis that goes beyond generic SSH troubleshooting.

In the **Rocket.Chat** tab (log in with `{user}` / `{password}` if prompted), check `#support` for the notification.

== Takeaways

* Athena's agents and skills are defined in configuration, not code
* OpenShift provides the extensibility mechanism: ConfigMap for agents, PVC for skills
* Adding a new specialist agent is a 3-step process: create the skill, edit the ConfigMap, restart the pod
* Skills encode institutional knowledge in plain language — anyone who understands the domain can write them
* The `ops_manager` automatically routes failures to the right specialist based on the agent's `description` field
* You just extended an AI operations team without writing any Python
